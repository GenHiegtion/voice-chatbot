# Real SSE Streaming Implementation Plan

## 1) Current Status overview
- The system already has SSE for text chat at `/api/chat/stream`, currently streaming raw tokens directly from LangGraph.
- The other APIs are batch-based: `/api/speech-to-text` and `/api/voice-chat` (upload a file and wait for the result).
- Current SSE does not have a clear event schema (tokens are just raw data strings), does not return `action/action_data`, lacks heartbeats, and does not handle disconnects gracefully.

## 2) Objectives
- Implement true standard SSE streaming: structured event types, JSON payload data, heartbeats, and clear termination events.
- Unify the stream schema across relevant APIs, prioritizing the real-time experience for chat and voice chat endpoints.
- Do not break existing backward compatibility (keep the standard `/api/chat` and `/api/voice-chat` endpoints as they are).

## 3) APIs to Upgrade with SSE
1) `/api/chat/stream` (existing) — upgrade to "real SSE streaming".
2) `/api/voice-chat/stream` — highly suitable since voice chat involves ASR + LLM + Chat steps, we can stream both progress indicators and chat tokens.
3) `/api/speech-to-text/stream` (optional) — if the ASR model supports partials, we can stream transcripts; if not, we can stream progress events and the final result.

## 4) Proposed SSE Event Schema Design
Every event will send data in JSON format so clients can parse them consistently:

- `event: meta`
  - `data: {"session_id": "...", "request_id": "..."}`
- `event: status`
  - `data: {"stage": "routing|answering", "node": "..."}`
- `event: progress`
  - `data: {"stage": "asr|correction|chat", "status": "start|done"}`
- `event: token`
  - `data: {"text": "..."}`
- `event: final`
  - `data: {"response": "...", "action": "UPDATE_CART|None", "action_data": [...]}` 
- `event: error`
  - `data: {"message": "..."}`
- `event: done`
  - `data: {"done": true}`
- `event: ping` (heartbeat)
  - `data: {"ts": 1714630000}`

Notes:
- For `/api/chat/stream`: meta → status (routing/answering) → token → final → done.
- For `/api/voice-chat/stream`: meta → progress(asr start/done) → progress(correction start/done) → status → token → final → done.

## 5) Phased Implementation Plan

### Phase 1 — Standardize SSE for Text Chat (Priority)
- Refactor `/api/chat/stream` to emit correctly formatted events with type + JSON (token, final, done, error).
- Include `action/action_data` in the `final` event (currently lost during stream).
- Add periodic heartbeats (e.g., every 15–30s) to keep the connection alive.
- Handle disconnections: if the client disconnects, stop the generator to reduce server load.
- Filter out JSON routing tokens generated by router nodes (coordinator, data_team, action_team) to keep the stream clean.

### Phase 2 — Add `/api/voice-chat/stream`
- Create a new endpoint for voice chat SSE.
- After ASR processing finishes, stream progress events, then begin streaming LLM tokens just like `/api/chat/stream`.
- The `final` event should include the transcript (`original_text`, `corrected_text`) alongside the `response` and `action/action_data`.

### Phase 3 — (Optional) `/api/speech-to-text/stream`
- If ASR supports partials: stream transcript segments.
- If not: stream progress + final result only.

### Phase 4 — Documentation & Client Examples
- Update README or API docs to describe the SSE schema and provide curl examples.
- Provide a client example (JS) on how to read events, concatenate tokens, and handle the final/action states.

## 6) Expected Technical Changes
- Create a shared SSE module (e.g., `src/api/sse.py`) to handle:
  - JSON event formatting
  - Heartbeat loops
  - Disconnect detection helpers
- Update the chat router and add the voice-chat/stream router.
- Add tests:
  - `test_chat_stream_schema`
  - `test_voice_chat_stream_progress`
  - `test_stream_error`

## 6.1) Required SSE Headers
- `Content-Type: text/event-stream`
  - Evaluation: Mandatory. This is the standard identifier for SSE, allowing clients to parse it correctly.
- `Connection: keep-alive`
  - Evaluation: Reasonable with HTTP/1.1 to keep the connection open. Note that this header is forbidden in HTTP/2; it can be set conditionally or dropped if running strictly over HTTP/2.
- `Cache-Control: no-cache`
  - Evaluation: Mandatory to prevent intermediate proxies from caching old data, ensuring real-time streaming.
- `X-Accel-Buffering: no`
  - Evaluation: Recommended if running behind Nginx to disable buffering, preventing data from being batched before being pushed out.

## 7) Risks & Considerations
- SSE traversing through proxies requires buffering to be disabled (handled by `X-Accel-Buffering: no`). Nginx/ALB must have sufficiently long timeouts.
- Token streaming depends on events from LangGraph; need to accurately filter by node name (`menu_agent`, `order_agent`, `promotion_agent`, `agent`) to prevent router JSON leakage.
- For voice chat: It remains a batch audio upload. True real-time speech would require WebSockets or chunked uploads (proposed for future upgrade).

## 8) Completion Criteria
- `/api/chat/stream` returns fully formatted JSON events with types, including `final` + `action/action_data` and filters out routing garbage.
- `/api/voice-chat/stream` is operational, allowing clients to observe progress + tokens + final results.
- Tests pass and documentation is up-to-date.
